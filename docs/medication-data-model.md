# BP Tracker — модель даних прийому ліків (погоджена)
 
Підсумок Етапу 0: погоджена структура сутностей/таблиць. Деталізація чотирьох осей дозування й обґрунтування через FHIR — у `medication-dosage-model_2.md`. Тут — **які таблиці, які колонки, як зв'язані**.
 
Реалізація, яку описує цей документ: C#-класи + EF Core (`OwnsOne` для owned types), PostgreSQL 16. Зміни структури — через EF-міграції (`dotnet ef migrations add ...`), накочуються на старті. Це специфікація Етапу 0 з C#-проєкту, яку новий FastAPI-бек перекладав у власні модульні моделі, а не копіював дослівно (див. `PROJECT.md`). У трьох місцях переклад свідомо відійшов від рішень, зафіксованих тут нижче: обмеження на активні призначення, прив'язку `reminder_config` і форму `intake_reports`. Кожне з цих трьох місць позначено нижче приміткою «Переглянуто у FastAPI-беку» з поясненням, яке рішення діє зараз і чому. **Актуальну схему для FastAPI-бекенду веде `README_DEV.md`** — якщо цей документ і `README_DEV.md` колись розійдуться знову, чинним вважати `README_DEV.md`.

## Ключовий принцип
 
**Проєкція — вперед, снапшот — назад.**
- Майбутні прийоми («сьогодні/найближче») **обчислюються** з призначення + годин нагадувань. Це не таблиця.
- Підтверджений факт прийому зберігає **власну копію** прийнятого і не залежить від поточного стану призначення.
Наслідок: **розклад похідний від призначення. Немає призначення — немає розкладу.**
 
ReminderTemplate (стара сутність, що дублювала ліки зі схеми) — **схлопнуто**. Від нього лишились тільки години слотів + параметри нагадувань, які тепер живуть у `reminder_config`.

> **Переглянуто у FastAPI-беку.** Тут і в таблиці `reminder_config` нижче йдеться про `reminder_config` як 1:1 дитину призначення (`prescription_id` як PK, каскадне видалення разом із призначенням). У FastAPI-беку це рішення замінено: `reminder_config` — один рядок на користувача (`user_id` як PK), не на призначення. Причина: години нагадувань — властивість людини, а не рецепта, і єдина форма, сумісна з тим, що в користувача може бути кілька одночасно активних призначень (див. нижче). Деталі й контракт — у `README_DEV.md`.

### reminder_config
Години слотів + параметри нагадувань. У цьому документі — **1:1 дитина призначення**: видалили призначення → каскадом пішов конфіг і весь розклад. У FastAPI-беку ця прив'язка інша, див. примітку нижче.
| Колонка | Тип | Ключ | Примітка |
|---|---|---|---|
| ~~prescription_id~~ / `user_id` | uuid | PK + FK | у FastAPI-беку — `user_id` (FK → `users`), один рядок на користувача, не на призначення |
| morning_time | time | | напр. 08:00 |
| day_time | time | | напр. 14:00 |
| evening_time | time | | напр. 20:00 |
| max_reminders | int | | |
| duration_minutes | int | | вікно підтвердження |

> **Переглянуто у FastAPI-беку.** PK і FK цієї таблиці — `user_id`, а не `prescription_id`; каскад іде від видалення користувача, не призначення. Обґрунтування — у примітці до «Ключового принципу» вище.
 
### medication_items
Позиції ліків. **Окрема таблиця**, one-to-many до призначення через FK.
Один рядок = один препарат у рецепті.
| Колонка | Тип | Ключ | Примітка |
|---|---|---|---|
| id | uuid | PK | сутність із власним життям |
| prescription_id | uuid | FK → prescriptions | композиція (каскад) |
| medicine | string | | |
| condition | string | | напр. «після їжі» |
| when_slots | jsonb | | масив слотів, напр. `["Morning","Evening"]` |
| dose_amount | string | | **owned (dose)** — `"0.5"`, `"1"` |
| dose_unit | string? | | **owned (dose)** — опц.; якщо нема, одиниця в назві |
| freq_count | int | | **owned (frequency)** — скільки разів |
| freq_period | int | | **owned (frequency)** |
| freq_period_unit | string | | **owned (frequency)** — h / d / wk |
| course_type | string | | **owned (course)** — ongoing / course |
| course_start | timestamptz? | | **owned (course)** — для курсу |
| course_intakes | int? | | **owned (course)** — кількість прийомів (не календарні дні) |
 
> Колонки `dose_*`, `freq_*`, `course_*` — це EF owned types (`OwnsOne`): у C# окремі класи `Dose`/`Frequency`/`Course` для зручності, фізично — колонки рядка `medication_items`. Власного id/FK не мають, не існують без позиції. У `medication-dosage-model_2.md` це «чотири незалежні осі».
 
### intake_reports
Журнал прийомів. **Незмінний снапшот** — несе власну копію прийнятого.
| Колонка | Тип | Ключ | Примітка |
|---|---|---|---|
| id | uuid | PK | |
| user_id | uuid | FK → users | |
| ~~prescription_id~~ | ~~uuid?~~ | ~~FK → prescriptions (nullable)~~ | у FastAPI-беку відсутня, див. примітку нижче |
| period | slot | | Morning / Day / Evening |
| date | date | | з урахуванням таймзони клієнта |
| ~~status~~ | ~~enum~~ | | у FastAPI-беку відсутня, див. примітку нижче |
| ~~time~~ / `taken_at` + `recorded_at` | timestamptz | | у FastAPI-беку — пара полів, див. примітку нижче |
| snapshot | jsonb | | копія на момент confirm: `[{medicine, amount, condition}]` |
 
> Унікальність у цьому документі — `(user_id, prescription_id?, period, date)`. У FastAPI-беку — `(user_id, period, date)`: `prescription_id` з ключа прибрано разом із самою колонкою.
> «Що прийнято» читається зі `snapshot`, **не** join'ом до поточного шаблону. Тому зміна/видалення призначення не псує історію. Це залишається чинним і в FastAPI-беку.
>
> **Переглянуто у FastAPI-беку** (три пов'язані зміни):
> - **`prescription_id` прибрано повністю**, не лишено навіть як `nullable`-трасування. Оскільки в користувача можуть бути кілька одночасно активних призначень (див. примітку до `prescriptions` вище), `snapshot` збирається з УСІХ активних призначень на момент запису, а не з одного — тож єдиний `prescription_id` як посилання «про яке це призначення» був би нечесним.
> - **`status` (`Confirmed`/`Missed`) прибрано.** «Пропущено» ніколи не зберігається рядком: рядок `intake_reports` існує лише якщо прийом підтверджено; «пропущено»/«вчасно»/«запізно» — проєкція на читання, яку завжди можна порахувати з уже збережених чесних моментів часу, а не збережений факт.
> - **`time` замінено на пару `taken_at`/`recorded_at`.** `taken_at` — коли ліки реально прийняті (клієнт надсилає явно, якщо момент відрізняється від «зараз»); `recorded_at` — коли рядок востаннє записаний у БД (завжди серверний `now()`, і при створенні, і при редагуванні). Одне поле `time` змішувало ці два різні факти.
>
> Повний контракт полів і поведінки — у `README_DEV.md`.

## Зв'язки (по FK)
 
- `users` 1 — * `measurements`, `prescriptions`, `intake_reports` — через `user_id`
- `prescriptions` 1 — * `medication_items` — через `prescription_id`; композиція (каскад)
- ~~`prescriptions` 1 — 1 `reminder_config` — через `prescription_id`; розклад похідний~~ — переглянуто у FastAPI-беку: `users` 1 — 1 `reminder_config`, через `user_id`

## Що НЕ таблиця
 
**Розклад «сьогодні/найближче»** — обчислювана проєкція, не сутність:
 
```
проєкція = medication_items УСІХ активних призначень користувача
           × години з reminder_config користувача
           (розгорнуті по слотах і, за потреби, по частоті/курсу)
```
 
Множина «УСІХ активних призначень» тут навмисна: у FastAPI-беку в користувача може бути кілька одночасно активних призначень (див. примітку до `prescriptions` вище), тож проєкція збирає позиції ліків з усіх них разом, так само як `snapshot` у `intake_reports`.
 
Точка обчислення одна — на беку. Для офлайн-алярмів натива бек матеріалізує плаский список на горизонт (натив кешує в Room, ставить AlarmManager сам).

## Відкрите питання
 
- **Глибина `course` у проді.** Зберігати `course_start`/`course_intakes` одразу, чи лишити позицію поки тільки з `course_type=ongoing`, а решту додати окремою міграцією під перший реальний курсовий кейс? Постійний прийом (Бісопролол) двигуна розмотування курсу не потребує. Сховище дороге в зміні — двигун дешевий; схиляємось зберегти поля, реалізувати двигун пізніше.
