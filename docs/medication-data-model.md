# BP Tracker — модель даних прийому ліків

Цей документ описує структуру сутностей і таблиць бази даних для призначень ліків, конфігурації нагадувань та журналу прийомів у FastAPI-бекенді. Деталізація чотирьох осей дозування й обґрунтування через FHIR — у `medication-dosage-model_2.md`.

## ВИРІШЕНО: IntakeReport — самодостатній незмінний снапшот

> Ця проблема існувала в C#-беку і була виправлена в FastAPI-реалізації.
> `intake_reports.snapshot` (JSONB) зберігає повну копію прийнятих ліків
> `[{medicine, amount, condition}]` на момент підтвердження. Зв'язку з
> `prescriptions` немає — снапшот самодостатній і не залежить від змін
> у призначеннях.

---

## Ключовий принцип

**Проєкція — вперед, снапшот — назад.**
- Майбутні прийоми («сьогодні/найближче») **обчислюються** на основі активних призначень та годин нагадувань. Ці записи не зберігаються в окремій таблиці.
- Підтверджений факт прийому зберігається як **незмінний снапшот** (`intake_reports`), який копіює склад прийомів і не залежить від подальших змін у призначеннях.
- Розклад є похідним від призначень: немає активних призначень — немає розкладу.

**reminder_config** (години слотів та параметри нагадувань) є глобальним налаштуванням користувача (`user_id` є PK/FK), а не окремого призначення. Оскільки в користувача може бути кілька одночасно активних призначень ліків від різних лікарів, години нагадувань визначаються на рівні профілю користувача.

### intake_reports
Журнал підтверджених прийомів ліків. Незмінний снапшот на момент підтвердження.
Унікальність (Unique Constraint): `(user_id, period, date)`.
| Колонка | Тип | Ключ | Примітка |
|---|---|---|---|
| id | uuid | PK | |
| user_id | uuid | FK → users | |
| period | slot | | Morning / Day / Evening |
| date | date | | календарна дата прийому в таймзоні клієнта |
| taken_at | timestamptz | | коли реально прийнято ліки |
| recorded_at | timestamptz | | коли запис зроблено в БД |
| snapshot | jsonb | | копія прийнятого: `[{medicine, amount, condition}]` |

> Зверніть увагу:
> - Зв'язку з `prescriptions` (`prescription_id`) немає, оскільки прийом може об'єднувати ліки з кількох різних активних призначень користувача на момент запису.
> - Статус прийому (`Confirmed`/`Missed`, `is_late` тощо) не зберігається в базі. Запис у таблиці `intake_reports` існує лише за фактом підтвердження прийому. Пропущені прийоми обчислюються на етапі читання як різниця між плановим розкладом та фактичними записами.
> - `taken_at` передається клієнтом, тоді як `recorded_at` встановлюється сервером (`now()`).

## Зв'язки (по FK)

- `users` 1 — * `measurements`, `prescriptions`, `intake_reports` — через `user_id`
- `prescriptions` 1 — * `medication_items` — через `prescription_id` (каскадне видалення)
- `users` 1 — 1 `reminder_config` — через `user_id` (каскадне видалення)
- Зв'язку між `prescriptions` та `intake_reports` немає.

## Що НЕ таблиця

**Розклад «сьогодні/найближче»** — обчислювана проєкція на етапі запиту:
```
проєкція = medication_items УСІХ активних призначень користувача
           × години з reminder_config користувача
           (розгорнуті по слотах і, за потреби, по частоті/курсу)
```
Оскільки користувач може мати кілька одночасно активних призначень, проєкція збирає позиції ліків з усіх них. Точка обчислення знаходиться на бекенді. Для офлайн-сповіщень клієнт завантажує плаский список на певний період часу і планує сповіщення локально.

## Відкриті питання

- **Глибина `course` у проді.** Зберігати `course_start`/`course_intakes` одразу, чи лишити позицію поки тільки з `course_type=ongoing`, а решту додати окремою міграцією під перший реальний курсовий кейс? Постійний прийом (Бісопролол) двигуна розмотування курсу не потребує. Сховище дороге в зміні — двигун дешевий; схиляємось зберегти поля, реалізувати двигун пізніше.
