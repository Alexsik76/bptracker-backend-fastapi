# Рішення (DECISIONS)

## Як вести цей файл

**Що записуємо.** Рішення, наслідки яких виходять за межі одного модуля, або
рішення, свідомо відкладені в часі. Локальні деталі реалізації сюди не потрапляють —
їхнє місце в коді та промптах-файлах.

**Що не записуємо.** Вибір, який видно з коду й який ніхто не переглядатиме окремо
(назва змінної, форма запиту, порядок аргументів). Для вузьколокальних нагадувань,
прив'язаних до рядка, — коментар `# TODO(scope): ...` у коді.

**Формат запису.** `ID` (`D-NNN`, за зростанням, не перевикористовується) + заголовок,
далі поля:
- **Статус** — `accepted` | `deferred` | `superseded`.
- **Дата** — `YYYY-MM-DD` ухвалення.
- **Контекст** — одне-два речення: чому питання взагалі постало.
- **Рішення** — що вирішено.
- **Тригер** — лише для `deferred`: подія, за якої рішення переглядають.

**Оновлення.** Змінюване рішення переписуємо на місці. Якщо його скасовано новим —
статус старого стає `superseded`, у тілі лишається посилання на `ID` наступника;
запис не видаляємо (історія рішень має бути читомою з одного файлу).

## D-001 — Доставка пошти інлайн, за стабільним інтерфейсом `EmailSender`

- **Статус:** accepted
- **Дата:** 2026-07-03

**Контекст.** Церемонія magic-link має надіслати лист. Постало питання, чи вводити
одразу чергу з ретраями (`email_outbox`), чи слати синхронно.

**Рішення.** Точка розширення — інтерфейс `EmailSender` (протокол + SMTP-реалізація).
Доставка лишається інлайн, синхронним `await` у церемонії: для входу семантично коректна
відповідь — «доставлено зараз або помилка зараз», щоб користувач одразу дізнався про збій
і повторив запит. Черга змінила б цей контракт на «прийнято, доставимо колись», що для
одноразового токена входу небажано. Універсальним робимо інтерфейс, не реалізацію:
поява черги пізніше (див. D-002) підмінить реалізацію за незмінним викликом
`await email_sender.send(...)`, без правок у місцях виклику.

## D-002 — `email_outbox` (черга доставки з ретраями) відкладено

- **Статус:** deferred
- **Дата:** 2026-07-03

**Контекст.** C#-бек має чергу доставки (`email_outbox`, ретраї, вкладення) за взірцем.
Для одиночного листа входу це надлишок (див. D-001).

**Рішення.** Черги зараз не вводимо. Логіка ретраїв SMTP тимчасово живе всередині
`SmtpEmailSender`; при появі черги переїде в її шар — локальна зміна в одному класі,
контракти не зачіпає.

**Тригер.** Поява фічі експорту даних поштою — першого споживача, для якого асинхронна
доставка з ретраями та вкладеннями є природною вимогою (обсяг, доставлюваність).

## D-003 — Обмеження частоти запитів (rate limiting) відкладено до деплою

- **Статус:** deferred
- **Дата:** 2026-07-03

**Контекст.** Ендпоінт запиту magic-link — очевидна ціль поштового бомбардування;
парольний вхід — ціль перебору. Тимчасові лічильники в застосунку зараз лише ускладнили б
код без інфраструктури під них.

**Рішення.** Тимчасових механізмів не вводимо. Rate limiting проєктуємо цілісно перед
деплоєм: edge за IP + застосунок per-email/per-user, лічильники в Redis.

**Тригер.** Підготовка до продакшн-деплою. Ендпоінти запиту magic-link і парольного входу —
пріоритетні цілі захисту.

## D-004 — Автентифікація passkey — usernameless (discoverable credentials)

- **Статус:** accepted
- **Дата:** 2026-07-12

**Контекст.** WebAuthn допускає два потоки входу. Email-first: клієнт надсилає email,
сервер повертає `allowCredentials`, відфільтрований під цього користувача. Usernameless:
клієнт нічого не надсилає, платформа сама пропонує наявні passkey, сервер упізнає
користувача за `credential_id` з відповіді.

**Рішення.** Обрано usernameless. Ключі реєструються з `resident_key=REQUIRED`;
`generate_authentication_options` викликається з `allow_credentials=[]`.

**Мотивація.** Email-first вимагає відповідати на запит options ще до автентифікації,
тобто повертати різні відповіді для наявного й відсутнього користувача — це та сама
проблема перелічування облікових записів, яку в magic-link довелося маскувати завжди-202.
Usernameless усуває її структурно: сервер до перевірки підпису не знає, про кого йдеться.
Додатково це відповідає поточному UX passkey у браузерах.

**Наслідок.** Порожній `allow_credentials` у `start_authentication` — навмисний, не
недогляд. Повернення до email-first потребувало б перереєстрації ключів.
